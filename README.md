# lol — Lots Of Logs

> A must-gather inspector for OpenShift / ROSA.
> Named in the proud tradition of [`omg`](https://github.com/ryane/omg) and [`omc`](https://github.com/gmeghnag/omc). We regret nothing.

`lol` is a Bash CLI that wraps [`omc`](https://github.com/gmeghnag/omc) with persistent **named contexts**, automated **check runs**, **signature-based issue fingerprinting**, and an AI-ready **handoff document** generator. It is designed for support engineers and SREs who spend a lot of time digging through must-gather bundles.

> **Status:** Early development. Commands and file formats may change. Contributions welcome.

## How it works

```
lol/
├── lol                  # entry point — subcommand dispatcher
├── lib/
│   └── common.sh        # shared helpers: colors, context resolution,
│                        # omc_use(), match_signatures(), scrub_pii()
├── checks/              # one script per check area
│   ├── etcd.sh
│   ├── nodes.sh
│   └── pdbs.sh
└── signatures/          # known-issue fingerprint files
    ├── etcd-high-latency.sig
    ├── etcd-member-unhealthy.sig
    ├── node-not-ready.sig
    ├── node-oom-eviction.sig
    └── pdb-blocking-upgrade.sig
```

### Context system

`lol` stores session state in `~/.config/lol/`:

```
~/.config/lol/
├── context              # path to the active must-gather
├── active_context       # name of the active named context (if any)
└── contexts/
    └── <name>/
        ├── meta.env         # name, created/updated timestamps, current mg path
        ├── mg-history.log   # all must-gathers associated with this context
        ├── commands.log     # ad-hoc omc commands run during investigation
        ├── runs/            # timestamped output from each `lol check` run
        └── handoff.md       # generated by `lol ready-up`
```

There are two session modes:

| Mode | How to start | Ledger? |
|------|-------------|---------|
| **Anonymous** | `lol use <path>` | No — nothing is recorded |
| **Named context** | `lol -c <name> use <path>` | Yes — checks, commands, and mg history are all logged |

A named context remains active across shell sessions until you either start a new anonymous session (`lol use <path>`) or resume a different context (`lol context resume <name>`).

### Checks

Each file in `checks/` is a self-contained Bash script that:

1. Resolves the must-gather path (from argument or stored context)
2. Calls `omc_use` to point `omc` at that path
3. Runs its checks using `omc` commands
4. Calls `match_signatures` on any suspicious output to identify known issues
5. Exits `0` (pass), `1` (findings), or `2` (error)

Checks are invoked by `lol check` but can also be run standalone:

```bash
bash checks/etcd.sh /path/to/must-gather
```

### Signature fingerprinting

Signatures in `signatures/` are plain-text files that map regex patterns to known named issues. When a check finds something suspicious, it passes the output to `match_signatures`, which scans the relevant `.sig` files and prints the issue name, summary, and remediation steps if a pattern matches.

Example signature file:

```
NAME=etcd-high-latency
SEVERITY=warning
SUMMARY=etcd fsync latency is high, often caused by slow disk I/O on control plane nodes
PATTERN=slow fdatasync
PATTERN=etcdHighFsyncDurations
PATTERN=heartbeat.*timeout
REMEDIATION=Check disk I/O on control plane nodes. etcd is extremely sensitive to
disk latency. Consider moving the etcd data dir to faster storage (NVMe preferred).
```

### omc passthrough

`lol` forwards a set of `omc` subcommands directly, automatically setting the `omc` context from lol's active must-gather. When a named context is active, the command and its output are appended to `commands.log`.

```bash
lol get pods -n openshift-etcd
lol describe node worker-1
lol logs -n openshift-etcd etcd-0
lol alerts          # shorthand for: omc get alerts -A
lol top nodes
lol projects
```

### ready-up

`lol ready-up` assembles everything in the active named context — mg history, check runs, ad-hoc commands — into a structured Markdown document suitable for handing off to an AI (Claude Code, etc.) for further investigation. PII scrubbing is applied by default (IPv4 addresses, UUIDs, email addresses, cluster URLs).

## Installation

```bash
git clone https://github.com/thephilip/lol.git
bash lol/install.sh
```

The install script will:

1. Clone the repo to `~/.local/share/lol/` (overridable via `$LOL_INSTALL_DIR`)
2. Symlink `~/.local/bin/lol` → the clone (overridable via `$LOL_BIN_DIR`)
3. Warn if `~/.local/bin` is not in your `$PATH`

Because the install directory is a git clone, `lol upgrade` works out of the box.

**Uninstall:**

```bash
bash install.sh --uninstall
```

This removes the symlink and optionally the install directory and context data.

> **Note:** `lol` is a directory-based tool — `lib/`, `checks/`, and `signatures/` must live alongside the `lol` executable. Do not copy just the `lol` file on its own.

### Dependencies

| Tool | Required | Purpose |
|------|----------|---------|
| [`omc`](https://github.com/gmeghnag/omc) | Yes | Must-gather parsing (all check and passthrough commands) |
| `bash` ≥ 4.0 | Yes | Associative arrays, `[[ ]]`, etc. |
| `jq` | For some checks | JSON parsing |
| `gum` or `fzf` | No | Optional — richer TUI prompts |

## Usage

```bash
# Set the active must-gather (anonymous session — no ledger)
lol use /path/to/must-gather-dir

# Run all checks
lol check

# Run specific checks (comma or space separated)
lol check etcd
lol check etcd,nodes,pdbs

# Named context — everything is recorded
lol -c 04425034-CoolBug use /path/to/must-gather
lol check
lol get pods -n openshift-etcd
lol alerts
lol describe node ip-10-0-1-1.ec2.internal

# Customer uploaded a second bundle — add it to the same context
lol -c 04425034-CoolBug use /path/to/second-must-gather
lol check etcd

# Generate an AI handoff document
lol ready-up -o handoff.md

# See what's in the active context
lol status
lol context show

# List and resume saved contexts
lol context list
lol context resume 04425034-CoolBug

# List available checks
lol list
```

## Contributing

### Adding a check

Create `checks/<name>.sh`. The contract:

- First comment after the shebang: `# DESC: one-line description` (shown in `lol list`)
- Source `../lib/common.sh` at the top
- Accept an optional `$1` as the must-gather path; use `resolve_mg_path "${1:-}"` to fall back to the stored context
- Call `omc_use "$MG_PATH"` before running any `omc` commands
- Call `match_signatures "<category>" "$output"` on suspicious output
- Exit `0` = clean, `1` = findings present, `2` = error

```bash
#!/usr/bin/env bash
# DESC: short description of what this check looks for

set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
source "$SCRIPT_DIR/lib/common.sh"

MG_PATH="$(resolve_mg_path "${1:-}")" || exit 2
FINDINGS=0

omc_use "$MG_PATH"

header "My Check"
output="$(omc get something 2>&1)"
echo "$output"

if echo "$output" | grep -q "bad thing"; then
  finding "Bad thing detected"
  ((FINDINGS++)) || true
  match_signatures "mycheck" "$output"
else
  ok "All good"
fi

echo
[[ $FINDINGS -eq 0 ]] && ok "mycheck passed"
exit $((FINDINGS > 0 ? 1 : 0))
```

The new check is automatically discovered by `lol list` and `lol check` — no registration needed.

### Adding a signature

Create `signatures/<category>-<name>.sig`. The `<category>` prefix must match what the relevant check passes to `match_signatures`. Multiple `PATTERN` lines are OR'd — any single match fires the signature.

```
NAME=short-identifier
SEVERITY=warning|critical
SUMMARY=Human-readable description of the issue
PATTERN=grep -E compatible regex
PATTERN=another pattern (any match fires)
REMEDIATION=What to do about it
```

### Linting

```bash
shellcheck lol lib/common.sh checks/*.sh
```
